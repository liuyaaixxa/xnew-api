# Device Token Management Code Review Checklist

## Overview
本文档提供了设备令牌管理功能的代码审查检查清单，用于确保代码质量和一致性。

---

## 1. 代码结构

- [ ] 文件结构遵循项目分层架构 (Router -> Controller -> Service -> Model)
- [ ] 新文件放置在正确的目录中
  - `/model/device_token.go` - 数据模型
  - `/service/octelium_service.go` - 服务层
  - `/controller/device_token.go` - 控制器
  - `/router/api-router.go` - 路由注册

## 2. 数据模型

- [ ] DeviceToken 结构体定义正确
- [ ] GORM 标签正确配置
  - `json:"-"` 用于 Token 字段（敏感信息)
  - `json:"token_mask"` 用于公开显示
- [ ] 表名使用 snake_case 命名约定: `device_tokens`
- [ ] 索引正确创建 (user_id 索引)

## 3. 数据库兼容性

- [ ] 遵循 CLAUDE.md Rule 2: 数据库兼容性规范
- [ ] 使用 GORM 抽象方法，不使用数据库特定语法
- [ ] 避免使用 `AUTO_INCREMENT` 或 `SERIAL`
- [ ] 布尔值使用整数 1/0 (不是 true/false)
- [ ] 测试覆盖 SQLite, MySQL 和 PostgreSQL 场景

## 4. 安全性

- [ ] Token 字段不在 API 娡型中暴露 (`json:"-"`)
- [ ] 返回脱敏的 token_mask 而非完整 token
- [ ] 用户只能操作自己的设备令牌 (user_id 校验)
- [ ] 所有 API 鰃用需要认证
- [ ] 敏感信息不记录到日志中时使用脱敏形式

## 5. 错误处理

- [ ] 所有错误返回统一的 JSON 格式
  - `{"success": false, "message": "..."}`
- [ ] 错误消息支持国际化
- [ ] 适当的 HTTP 状态码 (400, 401, 403, 404, 500)
- [ ] 错误消息有意义且用户友好

## 6. 输入验证

- [ ] 设备名称 (name) 验证
  - 食非空检查
  - 长度检查 (<= 64 字符)
- [ ] Domain 格式验证 (可选)
- [ ] 分页参数验证 (p, size)

## 7. 国际化

- [ ] 前端翻译已添加到 `/web/src/i18n/locales/zh-CN.json` 和 `en.json`
- [ ] 后端翻译已添加到 `/i18n/locales/zh-CN.yaml` 和 `en.yaml`
- [ ] 所有用户可见文本支持国际化
- [ ] 翻译键键使用中文作为键

## 8. 测试覆盖

- [ ] 单元测试
  - `/service/octelium_service_test.go` 学盖 OcteliumService
  - `/controller/device_token_test.go` 覬盖控制器逻辑
- [ ] 测试覆盖
  - 正常流程 (创建、查询、删除)
  - 边界情况 (空值、超长值 逶不存在)
  - 权限控制 (跨用户操作)
  - Token 脱敏验证
- [ ] 多数据库兼容性测试

## 9. 日志记录

- [ ] 关键操作记录日志
  - Token 生成
  - Token 撤销
  - 错误情况
- [ ] 敏感信息不在日志中暴露
- [ ] 日志格式与项目一致

## 10. API 文档

- [ ] API 端点文档完整
- [ ] 请求/响应示例清晰
- [ ] 错误码和文档准确
- [ ] 数据模型有完整说明

## 11. 代码风格

- [ ] 遵循项目代码风格指南
- [ ] 使用 `common.Marshal` 和 `common.Unmarshal` (Rule 1)
- [ ] 使用 `github.com/stretchr/testify/require` 进行断言
- [ ] 测试使用 `gin.SetMode(gin.TestMode)`

## 12. 依赖管理

- [ ] 新依赖添加到 go.mod
- [ ] 依赖版本固定 (使用特定版本或版本范围)
- [ ] octelium-go SDK 依赖占处理正确

## 13. 璁童恢复

- [ ] 错误处理不使用 panic
- [ ] 数据库错误有适当的恢复机制
- [ ] 外部服务 (octelium) 故障有优雅降级

## 14. 性能考虑

- [ ] 分页限制最大 100 条
- [ ] 数据库查询使用索引
- [ ] 避免不必要的数据库查询

## 15. 咨询代码审查

- [ ] 代码审查 PR 郚包含
  - 描述实现内容
  - 关联 Issue
  - 测试计划
- [ ] 截图日期
- [ ] 宣布测试通过后合并 PR

---

## 审查流程

1. **自我审查**: 开发者完成上述检查清单
2. **同行审查**: 至少一名其他开发者审查代码
3. **测试验证**: 所有测试必须通过
4. **集成测试**: 在测试环境验证完整功能
5. **代码合并**: 满足所有审查意见后合并代码
